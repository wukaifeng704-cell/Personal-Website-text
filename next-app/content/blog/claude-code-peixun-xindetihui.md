## 为什么「工具很强」不等于「团队会用」

在企业培训现场，我见过最常见的落差是：讲师演示时全场惊叹，两周后一问——「还是复制粘贴为主」。不是大家不努力，而是缺少把工具嵌进工作流的**脚手架**与**心理安全边界**。

## 三段式落地心法

### 1. 先划边界：能做什么、不能碰什么

在引入 Claude Code 的第一周，我会带团队先写一页纸的「红线条款」：哪些仓库只读、哪些目录可改、哪些必须人工 Review。边界清晰，才敢放手试。

### 2. 再给脚手架：模板、命令、检查清单

把高频任务固化成仓库里的 `README` 片段与脚本入口，让新人不必从空白提示词开始。培训现场我会留一版「课堂同款」最小示例仓，保证回去能跑通。

### 3. 最后上复盘：每周 15 分钟就够

固定问三个问题：本周最省时间的用法是什么？踩坑在哪？下周要试的一个新习惯是什么？复盘比加课更能放大培训 ROI。

## 小结

AI 工具教程的价值，不在于堆功能，而在于**可复制的节奏**：边界 → 脚手架 → 复盘。你在团队里若也遇到「工具热、落地冷」，不妨从一页纸边界开始试起。

下面是一段示例代码块（展示代码高亮）：

```ts
// 培训课后「最小可运行」检查：本地能 build、能跑单测、能打开 PR
async function afterClassCheck(repo: string) {
  await run(`git clone ${repo} && cd ${repo}`);
  await run("npm ci && npm test");
  return "ready-for-team-pilot";
}
```
